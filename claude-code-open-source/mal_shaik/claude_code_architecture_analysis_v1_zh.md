## 我研究了 Claude Code 的架构，这样你就不用自己去看了

不是怎么用它，而是它是怎么造的。

如果你在做 AI 产品，Anthropic 的工程决策简直是 agent 设计的教科书级别。

以下是我学到的东西，以及你可以直接拿去用的部分。

觉得有用的话，转发一下呗。

大多数人看到 Claude Code，只看到一个终端里的聊天界面。

我看了源码，看到的是一个 11 层的 agent 编排平台，几乎解决了 AI 产品开发中所有的难题。

每一层都有一个设计决策，你都可以应用到自己的 AI 产品里。不管你是在做编码 agent、客服机器人，还是 AI 工作流工具。

我们来逐层拆解。

# 

1\. 入口点被极致优化了速度

Claude Code 的启动只要几毫秒。怎么做到的？

快速路径路由。像 --version 和 --daemon 这样的简单命令，在完整应用加载之前就被拦截了。不做任何多余的初始化。

并行预取。在 CLI 解析你的命令的同时，它已经在加载设置、检查认证、建立 TLS 连接、预连接 API 了。全部同时进行。

记忆化初始化。昂贵的初始化操作只运行一次，然后缓存起来。

你可以学的：如果你的 AI 产品有 CLI 或启动流程，别一上来就加载所有东西。对常见场景走快速路径。并行预取。记忆化昂贵操作。用户对启动时间的感知比你想象的敏感得多。

# 

2\. 他们从零开始造了一个自定义终端渲染器

Claude Code 没有用标准的终端 UI 库。他们基于 React 用 Ink 自己造了一个渲染器，包含：

用于终端的 Yoga flexbox 布局引擎

带高度缓存的虚拟滚动

通过 interned screen buffers 实现的增量 ANSI diff 输出

CSI u 输入解析，支持鼠标操作和文本选择

他们真的把 Web 渲染的概念搬到了终端里。

你可以学的：别以为默认的 UI 框架就够用了。如果你的 AI 产品有独特的交互模式（流式响应、工具输出、多 agent 视图），就投入精力搞渲染层。一个能好好处理流式输出的自定义 UI 是巨大的用户体验优势。大多数 AI 产品的流式输出都很卡顿，就是因为没在这里投入。

# 

3\. 对话循环是一个 async generator 状态机

这是产品的核心。REPL 加载工具、构建 system prompt、加载上下文，然后进入一个 async generator 循环：

for await (event of query({messages, systemPrompt, tools}))

每个事件都被实时处理和渲染。工具调用、文本增量、错误——全部通过一个统一的流。

query 引擎本身是一个 while(true) 循环，它会：

1\. 归一化消息 + 压缩上下文

2\. 构建 system prompt（静态 + 动态）

3\. 带流式调用模型

4\. 收集 tool\_use 块

5\. 执行工具

6\. 追加结果

7\. 循环直到 end\_turn

你可以学的：如果你在做任何带工具调用的 AI 产品，这就是标准模式。用 async generators 处理事件流。用状态机循环处理 agent 周期。把"归一化上下文"、"调用模型"、"执行工具"分开。大多数 AI 产品把这些全搅在一起，最后变成意大利面代码。Anthropic 把它们分得很清楚，效果显而易见。

# 

4\. 工具执行有智能并发

Claude Code 有 60+ 个工具。但聪明的地方在于怎么执行它们。

当模型返回多个工具调用时，系统会对它们分区：

并发批次——只读工具（文件读取、搜索、glob）并行运行

串行批次——变更工具（编辑、写入、bash）顺序运行

每个工具都经过：输入验证（Zod）→ 前置钩子 → 权限检查 → 执行 → 后置钩子 → 结果截断

你可以学的：如果你的 AI 产品用到工具，想想并发。读操作可以安全地并行。写操作需要有序。大多数 AI 产品全部串行执行，没必要地慢。另外：用 schema 验证输入。截断输出。你的模型不需要在上下文里放一个 50KB 的文件，8KB 就够了。

5\. 权限系统是一个多层竞赛

这真的是很聪明的工程设计。

当 Claude Code 需要某个操作的权限时，它不只是问用户。它会并行竞赛多个解析器：

用户点击（对话框）

hook 分类器（自动化规则）

bash 安全分类器（基于 LLM 的安全检查）

bridge/web UI（外部审批）

谁先响应谁赢。createResolveOnce 模式。

在此之上，还有一个 5 级规则级联：policy > flag > local > project > user。高级别的规则不会被低级别的覆盖。

你可以学的：如果你的 AI 产品做任何有风险的操作（文件访问、API 调用、数据修改），就建一个分层权限系统。别每次都弹窗问用户。要有可配置的规则、自动分类器和交互式兜底。竞赛模式特别聪明——它意味着最快的安全路径总是赢。

# 

6\. 上下文管理有 5 种压缩策略

从这里你能看出工程师花了几个月。

默认 200K token 窗口。可选 1M。还有五种策略管理溢出：

microcompact——根据时间清除旧的工具结果

context collapse——将对话片段总结成更短的版本

session memory——将结构化上下文（任务规范、文件、学习记录）提取到文件中

full compact——总结整个对话历史

PTL truncation——作为最后手段丢弃最早的消息组

系统按从最少损失到最多损失的顺序尝试。当上下文压力来了，它自动处理。

你可以学的：如果你的 AI 产品的对话超过几轮，你就需要一个上下文管理策略。大多数产品只是从头部截断。这是最蠢的方法。建分层压缩。把重要上下文保存到外部存储。先总结再截断。一个"丢失上下文"的产品和一个"记住所有东西"的产品之间的差别，就是这个工程。

# 

7\. system prompt 被拆分为可缓存和动态两部分

Anthropic 把 system prompt 分成两个部分：

静态（可缓存）：角色指令、工具指南、编码规则、风格规则。这些在不同轮次之间很少变化。

动态（每次请求）：CLAUDE.md 文件、环境信息、git status、当前日期、记忆。这些每轮都变。

它们之间有一个明确的缓存边界。静态部分由 API 缓存（1 小时 TTL）。动态部分每轮重建。

你可以学的：如果你在向 LLM API 发送 system prompt，用同样的方式拆分。把稳定的指令放前面（可缓存）。把动态上下文放后面。这能大幅降低 API 成本，因为缓存的前缀不会被重新处理。大多数 AI 产品每轮都重建整个 system prompt，每次都付全价。

# 

8\. 子 agent 为缓存共享而设计

当 Claude Code fork 一个子 agent 时，它创建一个字节级一致的父上下文副本。这意味着 fork 出来的 agent 命中和父 agent 相同的 prompt 缓存。

5 个并行 agent 的成本几乎不比 1 个串行 agent 多。

三种执行模型：

fork——同一进程，共享缓存，不允许递归 fork

teammate——独立的 tmux/iterm 面板，基于文件的 mailbox 通信

worktree——每个 agent 一个 git worktree，隔离的分支

你可以学的：如果你在做多 agent 系统，考虑缓存拓扑。共享上下文前缀的 agent 可以共享 API 缓存。设计你的 agent 派生方式以最大化缓存命中。另外：agent 之间基于文件的通信简单又可靠。别用消息队列过度设计，在已知目录里放一个 JSON 文件就行了。

# 

9\. hook 系统让它无需 fork 就能扩展

25+ 个生命周期事件。5 种 hook 类型（command、prompt、agent、HTTP、function）。hook 可以来自设置、插件、agent frontmatter 或 SDK 回调。

PreToolUse 可以在操作发生之前阻止或修改它。PostToolUse 可以转换结果。UserPromptSubmit 可以向每条消息注入上下文。

你可以学的：从第一天就在你的 AI 产品中构建 hook。即使你不马上用。在每个工具调用、每条消息、每个会话之前/之后运行自定义逻辑的能力——这就是把产品变成平台的关键。你的高级用户会在你的 hook 之上构建出你从未想象过的东西。

# 

10\. 一切都是持久化和可恢复的

对话保存为 JSONL。会话记忆提取到文件。会话可以继续、恢复或 fork。

你可以学的：持久化一切。对话、工具结果、agent 状态。让它可恢复。存储成本跟丢失上下文的成本比起来不值一提。大多数 AI 产品把每个会话当作临时的，这对长时间运行的任务来说是灾难性的用户体验。

# 

大局观

这个架构揭示了关于构建 AI 产品的什么：

• 把渲染层和 agent 逻辑分开。Claude Code 用 React 做 UI，用 async generators 做 agent 循环。它们完全解耦。这就是为什么他们能用同一个核心支持终端、web bridge 和 SDK 接口。

• 把上下文当作受管资源来对待，而不是垃圾场。存在 5 种压缩策略，是因为"把所有东西都发给模型"根本不可扩展。每个严肃的 AI 产品最终都需要上下文管理。趁早搞。

• 从第一天就为并行设计。工具批处理、子 agent fork、缓存共享。这个架构假设多件事情同时发生。单线程的 agent 循环是死路一条。

• 让权限可配置，而不是二元的。一个带自动分类器的 5 级级联比"全部允许"或"每次都问"高级太多了。你的用户有不同的风险偏好。让他们自己配置。

• hooks 优于 plugins 优于硬编码功能。Claude Code 最具扩展性的部分不是任何单个功能，而是让任何人都能在不碰核心的情况下添加功能的 hook 系统。

• 缓存感知的架构能省钱。拆分 system prompt 为静态/动态、跨子 agent 共享缓存、使用 ephemeral 标记。这些是烙印在架构中的成本优化。在规模化时，这是盈利和破产之间的区别。

给开发者的总结：

• 入口点走快速路径。不需要的别加载。

• 投入精力搞流式/渲染层。这就是用户体验。

• 用 async generator 状态机做 agent 循环。

• 分区工具执行：读操作并行，写操作串行。

• 竞赛多个权限解析器。最先的安全答案赢。

• 5 层上下文压缩，而不仅仅是截断。

• 拆分 system prompt：静态（缓存）+ 动态（每轮）。

• 围绕缓存共享设计子 agent 派生。

• 从第一天就建 hook。hook 把产品变成平台。

• 持久化一切。让会话可恢复。

Claude Code 不只是一个好产品。它是如何构建 AI 原生软件的蓝图。
