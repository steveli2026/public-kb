## 我读完了 Claude Code 的源码，这样你就不用自己读了

大多数人打开 Claude Code，敲一段 prompt，等回复，再敲一段 prompt。

这就像买了辆法拉利却只挂一档开。

我想搞明白为什么有些人用 Claude Code 的产出能比别人多 10 倍。所以我做了一件正常人都会做的事：把整个源码读了一遍（我说的"我"当然是指 Claude Code 自己）。

11 层架构。成千上万行代码。一个伪装成终端聊天的 agent 编排平台。

以下是源码揭示的关于如何真正用好这个工具的发现。

## 

1\. CLAUDE.md 每一轮都会被加载。每。一。轮。

这是你能做的最高杠杆的事，但几乎没人做对。

大多数人的 CLAUDE.md 要么是空的，要么写了一整本圣经进去。

源码表明 Claude Code 在每一次查询迭代时都会重新读取你的 CLAUDE.md 文件。不是在 session 开始时。是每一轮。也就是说你每发一条消息，它都会重新读取你的指令。

有一套完整的层级体系：

\- ~/.claude/CLAUDE.md — 全局配置（你的编码风格、偏好）

\- ./CLAUDE.md — 项目级（架构决策、代码规范）

\- .claude/rules/\*.md — 模块化规则

\- CLAUDE.local.md — 私有笔记（加入 gitignore）

你有 40,000 个字符的空间。这非常多。大多数人可能只用了 200 个。

把你的架构决策写进去。你的文件命名约定。你的测试模式。你的"绝对不要做这个"规则。模型每一轮都会读取它们。这就是 Claude Code 作为一个通用助手和作为懂你的代码库的专属助手之间的区别。

如果你看完这篇文章只做一件事，就做这个。

## 

2\. 子 agent 共享 prompt 缓存（并行基本上是免费的）

这是让我最震撼的一点。

当 Claude Code fork 一个子 agent 时，它会创建一个与父级上下文完全一致的字节级副本。API 会缓存这个。所以生成 5 个 agent 分别处理你代码库的不同部分，成本几乎和 1 个 agent 顺序处理一样。

再读一遍上面那句话。

5 个 agent。和 1 个的成本一样。因为它们都命中了 prompt 缓存。

大多数人把 Claude Code 当作单线程工人使用。一次一个任务。等它完成。再给下一个。

源码里有三种子 agent 的执行模型：

\- fork — 继承父级上下文，缓存优化

\- teammate — 在 tmux 或 iTerm 中独立窗格运行，通过基于文件的信箱通信

\- worktree — 获得独立的 git worktree，每个 agent 一个隔离分支

你可以让 Claude Code 同时启动 5 个 agent：一个做安全审计，一个重构认证模块，一个写测试，一个更新文档，一个修 bug。全部同时进行。全部共享缓存。

这个架构就是为此而生的。单线程使用简直是暴殄天物。

## 

3\. 权限系统是被设计来配置的，不是用来一直点"允许"的

每次 Claude Code 问你"允许这个操作吗？"然后你点允许，这是配置的失败，不是功能。

源码揭示了一个 5 级设置级联：

```text
policy > flag > local > project > user
```

  

在 ~/.claude/settings.json 中你可以设置 glob 模式来指定哪些操作始终允许：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(git *)", 
      "Edit(src/**)",
      "Write(src/**)"
    ]
  }
}

```

  

有三种权限模式：

\- bypass — 完全不做权限检查（危险但快速）

\- allowEdits — 自动允许在工作目录中编辑文件

\- auto（这是新的）— 用一个 LLM 分类器对每个操作做判断。这是最佳平衡点。

auto 模式有自己的允许/拒绝列表可以配置。源码显示它会并行竞速多个解析器：用户点击、hook 分类器、bridge，谁先响应谁赢。

每次你停下来点"允许"都是在浪费时间。配置一次就好。再也不用点了。

## 

4\. 有 5 种压缩策略。上下文压力是个真实的问题

源码有五种不同的方式来压缩你的对话（当对话太长时）：

1\. microcompact — 基于时间清理旧的工具结果

2\. context collapse — 对对话片段做摘要

3\. session memory — 将关键上下文提取到文件

4\. full compact — 对整个历史做摘要

5\. PTL truncation — 丢弃最早的消息组

这告诉你一件重要的事：上下文溢出是工程师花了大量时间解决的核心问题。

对你来说意味着什么：

\- 主动使用 `/compact`。不要等系统自动压缩然后丢失你关心的上下文。

\- 默认窗口是 200K token。但你可以通过使用 `[1m]` 模型后缀来选择 1M token。对于跨多文件的大型重构，这很重要。

\- 长 session 会积累 "session memory" — 结构化的任务规格、文件列表、工作流状态、错误和经验教训的摘要。这就是为什么恢复 session 比重新开始更好。

\- 大型工具结果会被存到磁盘，只有 8KB 的预览发送给模型。如果你粘贴了一个巨大的文件，模型可能只看到一小部分。保持输入精简。

那些从 Claude Code 中获益最多的人像在游戏里手动存档一样使用 `/compact`。保留重要的，清除不重要的，继续前进。

## 

5\. Hook 系统才是真正的扩展 API（25+ 个生命周期事件）

这是几乎没人知道的高级用户功能。

源码揭示了 25+ 个你可以接入的生命周期事件：

\- PreToolUse — 在任何工具执行之前运行

\- PostToolUse — 在任何工具执行之后运行

\- UserPromptSubmit — 当你发送消息时运行

\- SessionStart / SessionEnd — session 生命周期

\- 以及 20+ 个其他事件

  

有 5 种类型的 hook：

\- command — 运行 shell 命令

\- prompt — 通过 LLM 注入上下文

\- agent — 运行完整的 agent 验证循环

\- HTTP — 调用 webhook

\- function — 运行 JS

  

实际能做的事情举例：

\- 每次写文件前自动运行 linting

\- 每次编辑后运行测试

\- 自动将相关文档注入每个 prompt

\- 任务完成时发送 Slack 通知

\- 在代码发布前验证安全模式是否被遵守

  

`UserPromptSubmit` hook 特别疯狂。你可以给你发送的每一条消息注入 `additionalContext`。想象一下，自动将测试输出、最近的 git diff 或项目状态附加到每个 prompt，而不需要手动输入。

这才是在 Claude Code 之上构建自定义开发环境的方式。不是通过写更好的 prompt。而是接入系统本身。

## 

6\. Session 是持久化且可恢复的（别再从头开始了）

每个对话都以 JSONL 格式保存在

```text
~/.claude/projects/{hash}/{sessionId}.jsonl
```

  

源码支持：

\- \--continue — 恢复你的上一个 session

\- \--resume — 选择一个特定的历史 session

\- \--fork-session — 从过去的对话分支出去（我个人超爱这个）

session memory 提取会在压缩过程中保留关键上下文：任务规格、文件列表、工作流状态、错误和经验教训。

大多数人每次打开 Claude Code 都开一个新 session。这就像每小时关掉 IDE 再从头打开一样。你在做什么、什么失败了、学到了什么 — 全没了。

用 `--continue`。永远用。让上下文积累。让 session memory 随时间积累经验教训。源码里就有专门的基础设施来支持这个。用起来。

## 

7\. 工具系统运行 60+ 个工具，带有智能批处理

Claude Code 有 60+ 个内置工具。但有趣的是它如何运行它们。

源码将工具调用分为两类：

\- concurrent — 只读操作（读文件、搜索、glob）并行运行

\- serial — 修改操作（编辑、写入、bash 命令）逐个运行

这意味着当 Claude Code 需要读取 10 个文件来理解你的代码库时，它会同时读取全部 10 个。但当它需要编辑 3 个文件时，会逐个执行以避免冲突。

除了内置工具外，你还可以连接 MCP 服务器来添加更多工具。源码使用延迟加载。MCP 工具只在需要时才加载，所以连接 5 个 MCP 服务器不会拖慢每个请求。

还有 ToolSearch 用于延迟发现 agent 尚不知道的工具。

实用建议：如果你的工作流涉及外部系统（数据库、云服务商、CI/CD），给它们连接 MCP 服务器。架构会处理复杂性。你只管获得更多能力。

## 

8\. 流式架构意味着中断的代价很低

整个管道使用 async generator 逐个 yield 事件。按 Escape 键可以干净地中止当前流，而不会丢失之前的上下文。

这看起来很小但它改变了你应该如何使用 Claude Code。

不要等一个你知道方向错了的回复跑完。立刻中断并重新引导。源码就是为此设计的。你之前的上下文被保留。中断的回复被干净地丢弃。零惩罚。

把它想象成结对编程。如果你的搭档开始走错方向，你不会等他做完。你会说"其实，走这边。"同样的意思。

## 

9\. 重试系统比你想象的更精密

源码揭示了：

\- 10 次重试，带指数退避和抖动（500ms 基数）

\- 在 401/403 时自动刷新 OAuth token

\- 模型降级：如果 Opus 连续 3 次遇到 529 错误，会自动降级到 Sonnet

\- 90 秒流式空闲看门狗 — 如果流式传输卡住，会回退到非流式模式

\- persistent 模式有无限重试，最大退避 5 分钟

这意味着 Claude Code 被设计成可以一直运行。它能优雅地处理 API 抖动、速率限制和故障。你不需要盯着它。让它在后台运行，回来看结果就行。

  

总结：源码中最高杠杆的操作：

• 写一个真正的 CLAUDE.md → 每轮都加载。40K 字符。最高杠杆配置。

• 用子 agent 并行化 → fork 模式共享 prompt 缓存。5 个 agent ≈ 1 个的成本。

• 在 settings.json 中配置权限 → 永远消除点击疲劳。

• 主动使用 /compact → 存在 5 种压缩策略，因为上下文压力是真实的。

• 设置 hook → 25+ 个事件，5 种类型。这才是真正的扩展 API。

• 永远用 --continue 恢复 session → JSONL 持久化 + session memory = 累积的上下文。

• 连接 MCP 服务器 → 延迟加载意味着不用时零成本。

• 随意中断 → async generator 意味着重新引导零惩罚。

底线：Claude Code 是一个穿着终端 UI 外衣的 agent 编排平台。

那些从中获得 10 倍产出的人不是 prompt 写得更好。他们配置了它。他们让它并行化。他们接入了它的 hook。他们让上下文跨 session 积累。

源码让这一切显而易见。现在你知道它底层到底是怎么运作的了。

如果你觉得有帮助就转发吧 🫶
