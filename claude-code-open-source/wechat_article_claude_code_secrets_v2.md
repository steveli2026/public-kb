# Claude Code 爆火后，我发现大家都把它用错了

## 备选标题

- 不是 prompt 不行，是你把 Claude Code 当聊天工具了
- 看完 Claude Code 源码分析，我更确定了一件事：会配置的人，效率真的高一个量级
- Claude Code 真正厉害的，不是写代码，而是这 5 个大多数人没用上的能力

## 正文

最近，围绕所谓“Claude Code 源码泄漏”的讨论很热。

但我把两篇分析看完之后，真正让我在意的，不是“源码到底泄没泄”，而是另一件事：

**很多人一直把 Claude Code 当成一个会写代码的聊天框。**

这恰恰是最可惜的地方。

因为从底层设计来看，它根本不只是“你问一句，它答一句”的工具。

它更像一个可以配置、可以记忆、可以并行、还能自动化的 AI 工程系统。

说得再直接一点：

**普通用户是在和一个助手聊天，高手是在带一个 AI 团队干活。**

所以，真正拉开差距的，从来不是谁更会写 prompt，而是谁先学会把工具“配起来”。

今天这篇文章，我不讲太多源码细节，只讲 5 件最值得你立刻拿去用的事。

---

## 一、先别急着提需求，先把它变成“懂你项目的人”

很多人每天都在抱怨：

“它怎么又改歪了？”

“它怎么老是不按我的技术栈来？”

“为什么每次都要重新解释一遍项目规则？”

问题往往不是它不够聪明，而是你根本没把规则交给它。

Claude Code 有一个很高杠杆的能力：它会反复读取你的项目说明。也就是说，你完全可以把项目约束、代码规范、禁区、测试要求提前写清楚。

最简单的做法，就是认真写好 `CLAUDE.md`。

别把它写成空白文件，也别把它写成 3 万字作文。对公众号读者来说，我更建议你先写这 5 类信息：

- 项目目标：这个项目最重要的目标是什么
- 技术栈：必须用什么，不能用什么
- 代码规范：命名、目录、接口风格
- 测试要求：哪些改动必须带测试
- 禁止事项：哪些文件、依赖、配置不要乱动

你可以把它理解成：给新同事的项目入职手册。

**空的 `CLAUDE.md`，就相当于让一个新同事在完全不了解背景的情况下直接开工。**

这也是很多人用 Claude Code 总觉得“不稳定”的根源。

不是它不稳定，是你的规则没有被稳定地输入进去。

---

## 二、别再把它当单兵用，要开始“分工”

这是我看完分析后感受最强的一点。

很多人用 Claude Code 的方式是这样的：

“先帮我改代码。”

等它改完。

“再帮我补测试。”

再等它做完。

“最后帮我写文档。”

这其实还是人在用传统外包思路指挥 AI。

但 Claude Code 更适合的用法，不是“一件件排队”，而是“同时分工”。

比如同一个需求，你完全可以一次性让它拆成几路：

- 一路检查现有实现有没有安全风险
- 一路负责重构核心逻辑
- 一路补测试
- 一路整理文档和变更说明

最后再统一汇总。

这背后的工程细节确实很强，但对普通用户来说，你只需要记住一句话：

**不要只让它像一个实习生一样干活，要让它像一个小团队一样协作。**

如果你会这么用，Claude Code 给你的就不再只是“代码补全”，而是“任务推进”。

这也是为什么有些人会觉得它只是“还行”，而另一些人已经把它用成生产力杠杆。

---

## 三、能配置的就别反复点击，别把时间浪费在“允许”上

很多人用 Claude Code，用到最后最烦的不是模型不够强，而是：

它老来问一句，“这个操作允许吗？”

点一次没事。

点十次也能忍。

但如果你一天要点几十次，整个人的工作流就被切碎了。

这件事的本质，不是安全做得太严，而是你没有把常用规则提前配好。

真正高频、低风险的动作，应该尽量一次配置掉。比如常规读取、项目内改文件、常见的测试命令、常见的 git 查看命令。

这样做的好处非常直接：

- 该快的地方快起来
- 真正危险的动作再确认
- 你的注意力不会被一堆低价值弹窗打断

很多人以为，AI 工具提效靠的是“更聪明”。

其实有时候，提效靠的是“少打断”。

**流畅，本身就是生产力。**

---

## 四、不要每次都从零开始，你需要的是“延续感”

很多人有一个非常浪费的习惯：

每次打开 Claude Code，都是全新对话。

这意味着什么？

意味着你昨天讲过的背景、踩过的坑、确认过的方案，今天还得再讲一遍。

这和找了一个能力很强、但每天都失忆的同事合作，几乎没区别。

Claude Code 更适合的方式，不是永远开新会话，而是尽量延续已有上下文。

说白了：

**把它当成长期协作对象，而不是一次性问答工具。**

你做过的决策、失败过的尝试、已经跑通过的路径，本来就应该被保留下来。

一旦你开始这么用，你会很明显地感受到两件事：

- 重复解释变少了
- 它对项目的“熟悉感”变强了

这不是一个小优化，这是从“每次重新培训”到“持续协作”的区别。

---

## 五、真正高级的玩法，不是多写 prompt，而是把重复动作自动化

很多人研究来研究去，最后研究的是 prompt 怎么写得更花。

但从 Claude Code 的设计思路来看，更高阶的玩法其实是另一条路：

**把那些你每次都会做、又没什么创造性的动作，交给系统自动完成。**

比如：

- 改完代码自动跑一下相关测试
- 写文件前先检查格式
- 发起任务时自动附上最近的变更信息
- 做完后自动整理一份变更摘要

你会发现，真正让人上瘾的不是“它会写代码”，而是“它开始自己接住流程里的琐碎工作”。

这时候你和它之间的关系就变了。

你不再是一个人在不停地下指令。

你是在搭一套会自己运转的工作流。

这也是为什么我越来越觉得，AI 工具的上限，不取决于模型参数，先取决于你的工作流意识。

---

## 对普通用户最重要的结论，其实只有一句话

**会用 Claude Code 的人，并不是更会提问，而是更会配置。**

他们做的事情无非就这几类：

- 用 `CLAUDE.md` 提前写清规则
- 把任务拆开，并行推进
- 把高频低风险动作配置掉
- 尽量延续 session，不反复重来
- 把重复动作做成自动化

你会发现，真正的差距不是“AI 会不会写代码”。

而是你有没有把它从一个聊天工具，升级成你的工作系统。

---

## 如果你是做 AI 产品的人，这里面还有 3 个特别值钱的启发

第一，**上下文不是垃圾场，而是资源。**

谁能更稳地管理上下文，谁的产品体验上限就更高。

第二，**并发不是锦上添花，而是能力放大的关键。**

很多产品输，不是输在模型不够强，而是输在流程还是单线程。

第三，**真正的扩展性，不靠硬编码功能堆砌，而靠可配置的规则、Hook 和工作流。**

这也是为什么有的 AI 产品越用越顺手，有的产品越用越像一次性玩具。

## 最后一句

如果你最近正好也被这波 Claude Code 讨论刷屏，我最想提醒你的不是“赶紧去围观源码”，而是：

**别再把它当聊天工具了。**

你只要先做 3 件事，体验就会立刻不一样：

1. 写一个真正有内容的 `CLAUDE.md`
2. 下次别开新会话，尽量延续上下文
3. 尝试把一个任务拆成多路并行推进
