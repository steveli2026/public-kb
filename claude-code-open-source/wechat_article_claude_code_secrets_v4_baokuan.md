# Claude Code 这波热闹里，我更在意另一件事

## 备选标题

- 同样都在用 Claude Code，为什么有人越用越顺，有人越用越累
- 很多人不是不会用 Claude Code，而是一直把它用在了错误的层级上
- 我看完 Claude Code 那两篇分析后，觉得大家讨论偏了

## 正文

这两天，很多人盯着“Claude Code 源码泄漏”这几个字看。

但我把那两篇分析读完，注意力反而没停在“泄漏”上。

我更在意的是另一个现象：

同样都在用 Claude Code，有人觉得它“挺强，但也就那样”，有人已经明显离不开了。

前者把它当成一个会写代码的对话框，后者已经把它放进自己的工作流里了。

这两种用法，表面看起来差不多，实际完全不是一回事。

很多人现在的使用方式，本质上还是“临时抓个助手干活”。

想到什么，就发一句。

发现偏了，再补一句。

做错了，再纠正一句。

下一次重开，继续从头讲背景。

这当然也能用，但它的上限不会太高。因为你始终是在用“口头交代”的方式带它干活。

你把规则放在嘴上，它就只能一边猜一边改。你每次都临时想起一点，它就每次都重新适应一点。

所以很多人觉得 Claude Code 有时候灵，有时候不灵。

在我看来，这往往不是模型忽然变笨了，而是你的协作方式一直没稳定下来。

---

## 真正该先补的，通常不是 prompt，而是项目规则

很多人遇到 Claude Code 跑偏，第一反应是重写 prompt。

其实不少时候，问题根本不在 prompt，而在于项目规则没有被稳定地交给它。

这也是为什么我越来越觉得，`CLAUDE.md` 这种东西不是可选项，而是基础设施。

你可以不把它写得很长，但至少该把这些事情说清楚：

- 这个项目到底优先什么
- 现在用的技术栈是什么
- 哪些地方能动，哪些地方别碰
- 改完之后最起码要做什么验证

这不是在“优化提示词”，这是在减少误解。

很多人把 Claude Code 当作一个很聪明的陌生人来用，所以每次合作都像第一次见面。

但你只要把最关键的规则提前放进去，它的表现就会稳定很多。不是因为它突然更聪明了，而是因为你终于不再让它反复猜你的边界。

---

## 另一个常见问题，是把所有事情都排成一条线

我发现很多人虽然在用 AI，但工作方式还是很传统。

比如一个需求过来，脑子里的顺序永远是这样的：

先改代码，再补测试，再写文档，最后再看看有没有别的问题。

这套顺序本身没错，但它默认了一件事：所有事情都得排队。

问题是，Claude Code 并不只适合“按顺序一个个做”。

不少事情本来就可以拆开。

有人负责看现有实现有没有坑，有人负责动核心逻辑，有人补测试，有人整理变更说明。你最后再统一汇总，和一条线慢慢往前推，体感完全不一样。

所以我现在很少把它理解成“一个能干活的助手”。

更准确一点，它像一个可分工的执行系统。你怎么拆任务，决定了它最后像不像生产力工具。

很多人花很多时间打磨一句话怎么说更漂亮，却很少花时间想一件更重要的事：

**这个任务到底该不该拆。**

---

## 还有不少低效，其实不是能力问题，是噪音太多

这点特别真实。

很多人用 Claude Code 用到最后，不是被难题卡住，而是被碎事磨没耐心。

动不动就要确认一次。

改几处文件，停一下。

跑个常规命令，再停一下。

回过神来，真正花在任务上的时间可能没多少，注意力倒是被切得很散。

这种低效很容易被忽略，因为它看起来不算“大问题”。

但实际工作里，最伤节奏的往往就是这种细碎打断。

所以我现在越来越觉得，AI 工具的体验不只看它聪不聪明，也看它顺不顺手。

规则该提前配的配掉，高频低风险的动作尽量别反复打断，真正危险的操作再卡住确认。这样工作流才像工作流，不然就像一路走一路踩刹车。

---

## 别老把上下文清零，这件事比很多人想得重要

还有一种很浪费的用法，就是每次都从新会话开始。

这样做最直接的代价，不是多打一段字，而是协作关系一直建立不起来。

你前面试过什么，否掉过什么，为什么这么改，哪些路已经走不通，这些本来都应该成为后续工作的背景。

如果每次都清零，它就始终停留在“单次响应”层面，很难真正进入项目状态。

这也是为什么有些人会觉得，Claude Code 用久了以后才开始顺。

不是因为你突然学会了什么秘技，而是因为上下文终于开始累积了。

你不再每次都从第一句讲起，它也不再每次都像刚入职那天。

---

## 真到这一步，自动化才会开始显得值钱

很多人一提到 AI，先想到的还是“怎么提问更高级”。

但看完 Claude Code 这些设计之后，我反而更确定：真正能拉开差距的，往往不是多会提问，而是多会把重复动作交出去。

比如改完代码顺手做的检查，比如一些固定的校验，比如每次都要补的上下文信息。这些事情单看都不大，但天天做、次次做，累积起来非常消耗人。

如果它能稳定接住这些动作，你的体验会发生一种很微妙但很重要的变化：

你不再觉得自己在“使用一个回答问题的工具”，而是在和一套已经磨合过的流程合作。

很多 AI 产品第一次用都让人觉得惊艳，真正留下来的却不多。

我现在越来越觉得，差别就在这里。

会回答问题，不够。

能进入工作流，才有黏性。

---

## 所以这波讨论里，我真正想说的是

如果你最近也在看 Claude Code 这件事，我觉得最值得带走的不是“又多知道了一个热点”。

而是一个很实际的提醒：

别总盯着 prompt 这一层。

很多时候，决定你用得顺不顺的，不是某一句话写得漂不漂亮，而是下面这些更基础的东西有没有搭起来：

- 项目规则有没有提前写清楚
- 任务有没有拆到合适的粒度
- 高频动作有没有减少打断
- 上下文有没有尽量延续
- 重复流程有没有慢慢自动化

这些事情都不新鲜，甚至有点“土”。

但真正能把工具用顺的人，往往就是把这些“土活”先做扎实了。

大家都在聊 Claude Code 强不强，我更想问的是另一句：

**你现在是在偶尔调用它，还是已经开始和它稳定协作了？**
