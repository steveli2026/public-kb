# 所有人都在用Claude Code，但只有1%的人知道它真正的样子

过去24小时，Claude Code源码"泄漏"刷爆了朋友圈。

我让Claude Code读了一遍自己的源码。11层架构，上千个文件，看完只有一个感受：

**99%的人在用法拉利送外卖。**

不是工具不行，是大多数人根本不知道这东西真正能干什么。

你以为它是个会写代码的聊天机器人。实际上它是个伪装成终端工具的agent编排平台。

差距有多大？大到你可能用了三个月，连40%的功能都没碰过。

今天就把这些被浪费的功能说清楚。

## CLAUDE.md这个文件，99%的人都是空的

先说个最夸张的。

Claude Code每一轮对话都会重新读取你的CLAUDE.md。不是session开始读一次，是每一轮。

你有40,000个字符的空间可以用。

但我打赌你的CLAUDE.md现在是什么样的？

空的。

这就是第一个巨大的浪费。

这个文件就是你和Claude Code之间的"工作协议"。你的编码风格、项目架构、测试标准、禁止事项——都应该写在这里。

写了之后是什么效果？

它不会再用any类型。它不会再把组件写成class。它不会再动package.json里的版本号。它记得你项目用的是Zustand不是Redux。

**每一轮都记得。**

大多数人的做法是什么？每次对话都临时补充："记得我们用的是Next.js啊"、"别用Redux啊"、"保持现有的命名风格啊"。

累不累？

写一次CLAUDE.md，这些话你永远不用再说。

项目根目录建个CLAUDE.md，写上：

```
# 项目规范

我们用Next.js 14 + App Router
状态管理用Zustand
永远不要用any
组件名PascalCase，文件名kebab-case
新功能必须写测试
```

就这么简单。别写成圣经，写成备忘录就行。

你项目的架构决策是什么？代码规范是什么？有什么坑需要避开？都写进去。

这是最高杠杆的配置，却是最被忽视的配置。

## 5个agent并行工作，成本几乎等于1个

第二个被浪费的功能更离谱。

Claude Code的子agent会共享父级的prompt缓存。

什么意思？**5个agent同时工作的成本，约等于1个agent串行工作。**

但大多数人怎么用的？

"帮我重构这个模块" → 等它做完 → "写一下测试" → 等它做完 → "更新文档" → 又等它做完

你在干什么？你在让法拉利走单行道。

这东西设计出来就是给你并行用的。

直接说："启动5个agent，一个做安全审计，一个重构认证模块，一个写测试，一个更新文档，一个修bug。全部并行。"

然后呢？

12分钟后5个任务全部完成。成本只比单线程多10%。

传统做法要50分钟。

这不是技巧，这是这个工具的基本设计逻辑。只不过99%的人不知道。

源码里有三种并行模式：fork、teammate、worktree。日常用fork就够了。缓存共享，几乎不增加成本。

你要是还在一个任务一个任务排队做，那真是暴殄天物。

## 权限系统不是让你点"允许"的，是让你配置的

第三个浪费。

每次弹出"允许这个操作吗？"，大多数人就点。点一次两次还好，点50次呢？

这不是安全机制，这是折磨。

很多人不知道的是：**这个权限系统从第一天就是设计来配置的，不是设计来点击的。**

打开`~/.claude/settings.json`，写上：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(git *)",
      "Edit(src/**)",
      "Write(src/**)"
    ]
  }
}
```

配完之后，这些操作永远不会再问你。

或者更狠一点，开auto模式。让LLM自动判断操作风险。只有真正危险的操作才会确认。

配置一次，永久解放。

但大多数人连这个文件的存在都不知道，每天还在那点"允许允许允许"。

你的注意力不是被难题打断的，是被这些低价值的确认框打断的。

## --continue这个参数，改变整个使用逻辑

第四个。

每次打开Claude Code都开新session的人，举个手？

这就像每小时关掉IDE重新打开一样。你之前做了什么、踩了什么坑、确认了什么方案——全没了。

Claude Code会把每次对话存成JSONL。它有专门的session memory系统，会提取任务规格、文件列表、错误教训。

这些都是可以跨session积累的。

但大多数人每次都`claude`一下，开个新的。

改成`claude --continue`。就这么简单。

你周一定的重构计划，周三继续做的时候，它还记得周一你们讨论的所有trade-off。

上下文会积累。经验会积累。你不用每次都从头解释项目背景。

这才是正常的工作方式。

## Hook系统，这才是真正的大杀器

第五个，也是最被忽视的。

Claude Code有25个生命周期事件。你可以在任何工具执行前后，插入自己的逻辑。

这才是把工具变成平台的关键。

举个例子。

你可以设置：每次写TS文件前自动跑ESLint修复，每次编辑代码后自动运行相关测试，每条消息自动附加最新的git状态。

全自动。你不用手动做任何事。

甚至你可以：代码提交前自动安全审计，任务完成时发Slack通知，每次对话自动备份。

这些都不需要改代码，不需要fork项目。在配置文件里写几行就行。

但有多少人知道这个功能？

5%都不到。

大多数人还在手动复制git status，手动跑测试，手动检查格式。

这些重复劳动本来就应该自动化掉。Hook系统就是干这个的。

你的高级用户会在hook之上构建出你从未想象过的工作流。

## 说到底，工具的上限是认知决定的

看完源码我最大的感受是什么？

**这些功能从第一天就在那。不是什么内测功能，不是什么隐藏彩蛋。**

CLAUDE.md从第一天就支持40K字符。
并行agent从第一天就支持缓存共享。
权限系统从第一天就可以配置。
Session从第一天就可以持久化。
Hook从第一天就有25个事件。

但99%的人不知道，或者知道了也不用。

为什么？

因为大多数人对工具的理解停留在"输入→输出"这个层面。

提个需求，等个结果。

但真正的高手在干什么？

**他们在配置工具，不是使用工具。**

他们会先花30分钟：
- 写CLAUDE.md，定义项目规则
- 配settings.json，消除重复点击
- 设hook，自动化重复工作
- 用--continue，积累上下文

配完之后呢？

效率直接上一个台阶。不是快一点点，是快一个量级。

这就是差距的本质。

不是谁prompt写得好。是谁先理解了这个工具的设计逻辑。

## 如果你在做AI产品，这份源码是教科书

看这个源码，普通用户能拿走用法。

做产品的人，能拿走的更多。

**上下文不是垃圾场，是需要管理的资源。**

Claude Code有5种压缩策略。从microcompact到full compact，从低损失到高损失依次尝试。不是简单粗暴地截断。

大多数AI产品怎么做？把所有东西塞给模型，超了就从头砍。

这就是专业和业余的差距。

**并发不是锦上添花，是基本能力。**

读操作全部并行，写操作串行执行。这是基本设计。

但你看市面上的AI产品，多少个工具调用还在串行执行？能并行的不并行，纯粹浪费时间。

**真正的扩展性是Hook，不是不断加功能。**

Claude Code最强的地方不是它有多少功能，而是它允许任何人在不碰核心代码的情况下扩展功能。

25个生命周期事件，5种hook类型。你想加什么逻辑，配置文件里写就行了。

这才是把产品变成平台的方法。

## 你现在还在"用工具"，还是已经在"带团队"？

回到最开始的问题。

为什么同样的工具，有人效率高一个量级？

不是因为他们prompt写得好。是因为他们已经不再把Claude Code当成一个"工具"在用了。

他们把它当成一个可以配置、可以记忆、可以并行、可以自动化的工作系统。

你给它规则，它按规则做事。
你给它任务，它拆开并行推进。
你配好权限，它不再打断你。
你写好hook，它自动接流程。

这才是Claude Code真正的样子。

现在问你一个问题：

**你是在"用一个聊天工具"，还是在"带一个AI团队干活"？**

想清楚这个问题，你的效率可能会直接翻倍。

不信你试试这三步：
1. 把CLAUDE.md从空白变成有规则
2. 下次启动用claude --continue而不是开新session
3. 在settings.json里配置高频操作的权限

就这三步，你的体验会明显不一样。

试完了来评论区说说感受。我很好奇有多少人的CLAUDE.md现在还是空的。
